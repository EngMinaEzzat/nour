Express-session is vulnerable to CSRF without a token. Install csrf-csrf (modern double-submit cookie for Express). Apply the middleware globally, exclude GET/HEAD/OPTIONS and Paymob webhook callbacks (they use HMAC instead).
Replit / LLM prompt

Expose a GET /api/csrf-token route that returns the token. Add the csrfProtection middleware to all routers except: health, paymob callbacks (POST /paymob/callback, POST /paymob/hmac-verify), and whatsapp callbacks. Update the frontend custom-fetch.ts to include the X-CSRF-Token header on every mutating request, fetching the token on app load and caching it in memory.

S4-T1Build TenantDb — Drizzle equivalent of TenantScopedPrismahigh
Port Project 1's elegant TenantScopedPrisma isolation pattern to Drizzle. Every table query should automatically include a tenantId WHERE clause to make cross-tenant leaks architecturally impossible.
Replit / LLM prompt
Create lib/db/src/tenant-db.ts that exports a class TenantDb. Constructor takes tenantId: number. It exposes typed query helpers for each tenant-scoped table. Example for products: products = { findMany: (where?) => db.select().from(productsTable).where(and(eq(productsTable.tenantId, this.tenantId), where)), findById: (id) => db.select().from(productsTable).where(and(eq(productsTable.id, id), eq(productsTable.tenantId, this.tenantId))).limit(1).then(r => r[0] ?? null), create: (data) => db.insert(productsTable).values({ ...data, tenantId: this.tenantId }).returning(), update: (id, data) => db.update(productsTable).set(data).where(and(eq(productsTable.id, id), eq(productsTable.tenantId, this.tenantId))).returning(), delete: (id) => db.delete(productsTable).where(and(eq(productsTable.id, id), eq(productsTable.tenantId, this.tenantId))) }. Implement this for all tenant-scoped tables: products, categories, orders, customers, merchants, discounts, returns, stockLogs, affiliates, reviews, cartSessions. Export TenantDb from the @workspace/db package. Add a factory: export const tenantDb = (tenantId: number) => new TenantDb(tenantId).

ground rules for the AI agent executing this plan

Rule 1 — typecheck must pass after every task

Run pnpm run typecheck after each individual task. If it fails, fix the errors before proceeding to the next task. Never leave the codebase in a state where typecheck fails.

Rule 2 — all user-facing strings must be Arabic

Every res.json({ error: "..." }) must contain Arabic text. English is only permitted in: code comments, console.log/warn/error, TypeScript type names, and variable names. Use src/lib/errors.ts (S6-T4) as the canonical error dictionary once created.

Rule 3 — never break existing passing tests

Before starting each sprint, run the test suite. After completing each sprint, run it again. If any previously-passing test now fails, fix it before marking the task done.

Rule 4 — environment variables must have fallbacks in dev

New env vars (REDIS_URL, VALU_API_KEY, etc.) must not crash the server when absent in development. Use graceful fallbacks or feature flags. Document every new env var in replit.md.

Rule 5 — tenant isolation is non-negotiable

No route may access tenant-scoped data without validating tenantId. If you're writing a query that touches products, orders, customers, or categories, it must go through TenantDb (after S4 is complete) or include an explicit eq(table.tenantId, req.merchantTenantId) WHERE clause.

Rule 6 — SSR catch-all must never intercept API routes

The SSR catch-all route in app.ts must be registered last and must explicitly skip any path starting with /api/. Otherwise it will shadow API routes and return HTML instead of JSON.

Rule 7 — do sprints in order, do not skip

Sprint 1 (security) must be fully complete before Sprint 2. Sprint 3 (SSR) depends on the API server being hardened first. Sprint 4 (TenantDb) must complete before any new route is added. Sprints 5 and 6 can be parallelised with each other but not with earlier ones.

Rule 8 — no magic strings, no hardcoded secrets

All external credentials (VALU_API_KEY, REDIS_URL, etc.) must come from environment variables. If a required variable is missing in production, fail fast with a descriptive error. Never fall back to a hardcoded secret or weak default.
